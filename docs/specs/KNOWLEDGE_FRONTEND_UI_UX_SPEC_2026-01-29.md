# 知识库（Knowledge）前端 UI/UX 设计规格（oMyTree）

**版本**：v0.2（修订稿）  
**日期**：2026-01-29  
**目标**：在不引入“双会话系统”的前提下，把「知识库管理」与「对话中引用知识库」做成 oMyTree 原生体验，并可直接进入实现。

## TL;DR（v0 结论）

1. 对话中知识库引用采用“圆形 @ 按钮 → 选择知识库（可选：再限制到文件）→ 输入框显示附件芯片”的模型。
2. v0 只支持“单知识库 + 可选文件限制”（不做多知识库融合），以降低交互复杂度并避免检索解释混乱。
3. 知识库管理入口放在树下新增“知识库”卡片；知识库本体按用户维度全局存在，可在树里“挂载/取消挂载”（v0 仅做挂载视图，跨树复用的 UI 先不做复杂化）。
4. 聊天会话统一走 `/api/turn` 与 `/api/turn/stream`；`/api/knowledge/sessions*` 不使用。

---

## 0. 背景与约束

### 0.1 现状
- oMyTree 已有成熟对话体系：`/api/turn`、`/api/turn/stream`（会话与多轮上下文统一由 turn 系统承担）。
- 知识库能力通过 oMyTree 适配层对外提供：`/api/knowledge/*`（内部转发 WeKnora）。
- `POST /api/knowledge/sessions` 与 `POST /api/knowledge/sessions/:id/chat` 已弃用（410），前端不得依赖。

### 0.2 产品约束（必须遵守）
- **用户永远不应感知 WeKnora session_id**；对话只认 oMyTree 的 turn。
- 知识库引用应是“对话输入的一个可见上下文约束”，而不是切换到另一套聊天产品。

### 0.3 范围边界（避免设计漂移）

v0 明确不做：
- 多知识库融合检索、rerank 质量评估、知识库共享/协作、复杂标签体系、在线编辑。
- 任何 WeKnora 原生 UI 的嵌入或品牌露出（用户看到的是 oMyTree 的知识库能力）。

v0 必须交付：
- 树内知识库管理（列表/创建/详情文档上传与列表/检索测试）。
- 对话中 @ 选择（KB + 可选 Doc 限制）与输入框附件芯片。
- 回答的引用来源展示（至少文档名 + 可展开片段）。

## 1. 设计目标（What good looks like）

1) **1 秒懂**：用户一眼知道“@=选择知识来源”，选了就变成输入框里的“附件芯片”。
2) **无割裂**：知识库管理在树内，但知识库可复用（按用户维度全局管理 + 可挂载到某棵树）。
3) **低操作成本**：90% 场景仅需“点 @ → 选知识库/文件 → 输入问题 → 发送”。
4) **可解释**：回答里能展示引用来源（文档名/片段），用户能点开查看。
5) **可扩展**：未来支持多个知识库、多个文件、多模式检索/重排，不推翻交互。

### 1.1 用户故事（v0 的验收视角）

- 作为用户，我可以在某棵树下创建一个知识库并上传文档，看到其处理状态最终变为可用。
- 作为用户，我在对话中点击 @ 选择该知识库，然后提问，回答优先基于该知识库内容并给出引用来源。
- 作为用户，我可以把范围进一步限制到某个文件，避免回答“发散到别的文档”。
- 作为用户，我在不选择知识库时，对话行为与现有体验一致。

## 2. 信息架构（IA）

### 2.1 入口
- **树主页**：在“新建树”下方增加一张卡片：**知识库**（可理解为“这棵树的资料柜”）。
  - 卡片展示：已挂载知识库数量 / 最近更新 / 入口按钮“管理”。
  - 点击进入：知识库管理页面（树域）。

#### 2.1.1 为什么入口在树下而不是全局侧边栏
- 用户的主要心智是“这棵树在讨论什么”；知识库作为树的上下文资产更自然。
- 技术上知识库可能是全局存储，但 UI 以树为视角可以最小化权限/共享讨论。

### 2.2 页面层级
- 树内：
  - `树 → 知识库（管理）`
  - `树 → 知识库 → 某知识库详情`（Documents / Search / Settings / Activity）
- 对话页：
  - 输入区新增 **圆形 @ 按钮**（位于模型选择胶囊右侧，和截图方案一致）
  - @ 弹窗选择“知识库/文件” → 以“附件芯片”方式附着在输入框

> v0：只做树内视角与“挂载到当前树”的筛选，不新增“全局知识库中心”页面，以免范围膨胀。

## 3. 关键交互：@ 选择 → 附件芯片 → 发送问答

### 3.1 组件位置与视觉
- **@ 圆形按钮**：放在“模型选择胶囊”右侧，尺寸与模型胶囊高度一致（或略小），保持同一行。
- 状态：
  - 默认：`@`（灰/中性色）
  - 已选择：`@`（高亮）+ 右上角小徽标显示已选数量（1…9+）
  - 禁用：在未登录/无权限时禁用，并给出 tooltip。

#### 3.1.1 行为约束（避免 UI 干扰输入）
- @ 弹层不得抢占输入焦点；用户选择后焦点回到输入框。
- 如果用户正在 IME 输入（中文拼写中），不得在键入 @ 字符时误弹出弹层（本设计明确由按钮触发，不绑定键入 @）。

### 3.2 @ 弹出层（Popover）布局
弹出层优先“快速选中”，结构如下：

- 顶部搜索框：支持按“知识库名/文件名/标签”搜索
- Tabs（可选）：`知识库` / `文件`
- 列表区：
  - **最近使用**（最近 5 个知识库/文件）
  - **已挂载到当前树**（更高优先级）
  - **全部知识库**
- 每项展示：名称、文档数量、最近更新时间、状态（可用/处理中）

操作规则：
- 点击知识库：选中该知识库（v0：单选）。
- 二级选择文件：在已选知识库下展示文件列表（v0：最多选择 20 个文件；超过给提示，避免 payload/检索负担）。
- 提供“清空选择”与“去管理”快捷入口。

#### 3.2.1 空/异常状态
- 无知识库：展示空状态 + CTA“新建知识库”。
- 知识库存在但无文档：提示“该知识库暂无可用文档”，并给“去上传”。
- 知识库处理中：允许选择，但发送时提示“部分文档仍在处理中，回答可能不完整”。

### 3.3 输入框内“附件芯片”规范
选中后在输入框左侧/上方出现 chips（如图三）：

- 芯片类型：
  - `KB 芯片`：例如“佛教”
  - `Doc 芯片`：例如“寺院信息化管理软件方案2024.docx”
- 芯片内容：图标 + 名称（超长省略） + 关闭（x）
- 多选策略：
  - **默认建议**：同一时刻允许 1 个 KB + 若干 Doc（Doc 必须属于当前 KB）
  - 高级模式（后续）：允许多 KB，并在发送时做合并检索
- 芯片交互：
  - hover 显示完整名称与基础元信息（大小/更新时间/解析状态）
  - 点击 Doc 芯片可展开“来源详情”（可选）

  #### 3.3.1 交互细节（可实现的最小集合）
  - 芯片区域最多显示两行；超出时折叠为“+N”，点击展开。
  - Backspace：当输入框为空时，Backspace 删除最后一个芯片（先 Doc 后 KB）。
  - 删除 KB 芯片会同时清空其下的 Doc 芯片。

### 3.4 发送问答的系统行为（产品层）
用户点击发送后：
- 若未选任何 KB/Doc：按普通对话走 turn。
- 若选择了 KB/Doc：系统把本轮问题标记为“基于知识库问答”，并保证：
  - 答案优先引用选中的 KB/Doc
  - 回答里可展示引用来源（可点击打开）

**推荐的后端契约（强制建议，避免前端拼 prompt）**：

- 在 `POST /api/turn` 与 `POST /api/turn/stream` 请求体里新增可选字段：
  - `knowledge`: `{ baseId: string, documentIds?: string[], topK?: number }`
- 服务端负责：检索 → 构造上下文 → 生成 → 输出 `citations`（至少包含 docId/docName/snippet 可选 score）。
- 服务端必须保证：不暴露 WeKnora 的 session/内部路径概念；前端只处理 oMyTree 语义字段。

> 如果短期无法改 turn API，也可做“前端编排版”：先调 `/api/knowledge/bases/:id/search` 获取 topK 片段，再把片段拼进 turn 的上下文；但这会带来 prompt 体积、可解释性与一致性问题，优先推荐服务端整合。

### 3.5 回答中的引用展示（v0 必做）

- 回答消息下方显示“引用来源”区域：
  - 默认折叠为 1 行：“引用 3 条来源”，可展开。
  - 展开后列表：文档名 + 可点击“查看片段”。
- 点击“查看片段”：弹出侧边抽屉/对话框显示 snippet（带高亮）与元信息（来自哪个 KB）。

> 目的：把“基于知识库回答”做成用户可验证的事实，而不是黑箱。

## 4. 知识库管理页面（树内视角）设计

参考 WeKnora 原版（图四之后），但做 oMyTree 原生化与更强的“树关联”。

### 4.1 页面 1：知识库列表（Tree → Knowledge）

#### 4.1.1 布局
- 顶部：页面标题“知识库”，副标题“管理并组织你的资料，用于对话问答”。
- 右侧主按钮：`新建知识库`
- 列表区：卡片网格（2~3 列）或表格（可切换）。

#### 4.1.1 v0 只保留一种呈现
v0 只做卡片网格（表格切换属于范围膨胀）。

#### 4.1.2 每个知识库卡片
- 名称、描述（1 行）、文档数、最近更新
- 状态：
  - 可用
  - 处理中（有文档解析/入库任务）
  - 异常（解析失败/向量化失败，提供重试入口）
- 快捷操作：`进入`、`挂载到当前树/取消挂载`（如果支持全局 KB）

#### 4.1.4 挂载定义（v0 UI 语义）
- “挂载到当前树”只影响：
  - @ 弹窗默认列表的优先级
  - 树下知识库卡片的计数与入口
- 不改变知识库的归属与权限（v0 不做共享，默认只对本人可见）。

#### 4.1.3 空状态
- 空：展示插画 + 文案“还没有知识库。新建一个，用 @ 在对话中引用。”
- CTA：`新建知识库` + `了解如何使用 @`

---

### 4.2 页面 2：新建/编辑知识库（Wizard）

采用左侧步骤导航（与 WeKnora 类似），但默认只暴露必要项：

步骤：
1) **基本信息**
   - 名称（必填）
   - 描述（可选）
   - 关联：是否挂载到当前树（默认勾选）
2) **分块设置**（建议默认折叠为“高级”）
   - chunk size（默认 512）
   - overlap（默认 100）
   - 分隔符（默认值集）
3) **检索/问答设置**（高级）
   - topK 默认值
   - 是否开启“引用来源”输出（默认开启）
4) **权限/共享（未来）**

保存：
- `创建知识库` / `保存`

#### 4.2.1 表单校验与反馈
- 名称：必填，长度 2-40，禁止纯空白。
- 创建成功：toast + 自动进入该知识库详情页（Documents Tab）。
- 创建失败：标准错误信封展示 message + hint。

---

### 4.3 页面 3：知识库详情（Knowledge Base Detail）

顶部：面包屑 `知识库 / {名称}` + 右侧操作（上传/导入/设置）。

Tabs：

#### Tab A：文档（Documents）
- 左侧：分类/标签（可选，v0 可不做）
- 右侧：文档列表（卡片或表格）

文档行展示：
- 文件名、类型、大小、状态、更新时间
- 状态细分：
  - 已入库
  - 解析中
  - 分块中
  - 向量化中
  - 失败（可查看原因/重试）

操作：
- 上传文档（file）
- 上传文件夹（批量）
- 导入网页（url）
- 在线编辑（可选/后续）
- 删除/重试/重新解析

##### v0 范围裁剪（建议）
v0 只做“上传文档（file）”与“删除”，其余入口先隐藏（避免做半套）。

#### Tab B：检索测试（Search Playground）
- 输入 query → 调用 `/api/knowledge/bases/:id/search`
- 展示结果：
  - 片段内容（高亮命中）
  - 分数 score
  - 来源文档名
- 提供“把该 query 发送到对话（携带 @）”的快捷按钮

##### v0 的价值
该页用于：验证入库是否成功、快速定位“检索没命中”的问题，是上线后排障必需品。

#### Tab C：设置（Settings）
- 基本信息
- 分块设置
- 高级：重建索引/重算向量（需二次确认）

#### Tab D：活动（Activity）
- 上传/解析/失败/删除等事件流（v0 可只做“最近任务”）

##### v0 建议实现
只做“最近任务”列表（最近 20 条），字段：时间/动作/对象/状态/失败原因。

## 5. 对话页与知识库管理的联动

### 5.1 “树内知识库”与“对话 @”的默认关系
- 用户在某棵树里对话时，@ 弹窗优先展示：
  1) 已挂载到该树的知识库
  2) 最近使用知识库
  3) 全部知识库

### 5.2 选择持久化建议
- “当前树的默认 KB”（可选）：当用户在该树中首次选择 KB 后，可记为默认，下次打开仍高亮。
- “最近使用”跨树共享：最近 5 个存用户维度。

#### 5.2.1 v0 建议
只做“最近使用（用户维度）”，不做“树默认 KB”，避免引入不可见的隐式状态。

## 6. API 对接映射（前端视角）

### 6.1 知识库管理
- 列表：`GET /api/knowledge/bases`
- 创建：`POST /api/knowledge/bases`
- 详情：`GET /api/knowledge/bases/:id`
- 更新：`PUT /api/knowledge/bases/:id`
- 删除：`DELETE /api/knowledge/bases/:id`

### 6.2 文档管理
- 上传：`POST /api/knowledge/bases/:id/documents/file`（multipart/form-data）
- 文档列表：`GET /api/knowledge/bases/:id/documents`

### 6.3 检索
- 检索：`POST /api/knowledge/bases/:id/search`

### 6.4 对话
- 非流：`POST /api/turn`
- 流式：`POST /api/turn/stream`

### 6.5 建议补充的 API（为前端体验兜底）

现有接口能跑通 v0，但为了减少前端推断与补请求，建议后端补充（或在现有返回中补字段）：
- 知识库列表返回：`status`、`updated_at`、`document_count`
- 文档列表返回：`status`（处理中/可用/失败）、`error_reason`（可选）、`mime/type`、`size`

> 这些字段用于前端做可靠的状态与错误展示，否则 UI 会出现“看起来成功但其实处理中”的体验缺陷。

## 7. 边界条件与错误体验

- 未登录：知识库入口隐藏或置灰（取决于产品策略），提示“登录后使用知识库”。
- 无知识库：@ 弹窗空状态 + CTA“新建知识库”。
- 文档解析失败：
  - 文档行显示失败原因（简短）
  - 提供“重试”“查看详情”
- 上传超限：提示大小上限（后端目前限制 50MB）。
- 检索无结果：显示“未在知识库中找到相关内容”，并建议“扩大范围/取消 Doc 限制/切换 KB”。

### 7.1 性能与退化策略

- @ 弹窗首次打开：必须在 300ms 内有可见反馈（哪怕是 skeleton）。
- 文档列表：默认分页/无限滚动（v0 可先 50 条分页）。
- 当网络错误时：
  - 不清空用户已选芯片
  - 允许用户重试发送

## 8. 可观测性（埋点建议）

关键事件：
- `knowledge.open_manager`（从树卡片进入）
- `knowledge.create_base` / `knowledge.update_base` / `knowledge.delete_base`
- `knowledge.upload_file`（成功/失败/耗时/文件类型）
- `chat.click_at`、`chat.select_kb`、`chat.select_doc`、`chat.clear_knowledge`
- `chat.send_with_knowledge`（是否流式、kbId、docCount、topK、回答耗时）

## 9. 可访问性与键盘操作

- @ 按钮可 Tab 聚焦；Enter 打开弹层；Esc 关闭。
- 弹层列表支持上下键选择、Enter 确认。
- 芯片支持 Backspace 删除最后一个。

### 9.1 额外要求
- 弹窗中每个列表项可被屏幕阅读器读出“名称 + 状态 + 文档数”。

## 10. 决策点（收敛为 v0 的明确结论）

1) **选择策略**：v0 = 单知识库 + 可选多文件限制。
2) **知识库归属**：v0 = 用户全局（数据层），树内只做“挂载视图”。
3) **turn API 扩展**：v0 = 接受新增 `knowledge` 字段，由服务端完成检索与 citations（不做前端拼 prompt）。
4) **引用展示**：v0 = 默认开启引用来源展示，且可展开查看片段。

---

## 11. v0 交付范围（建议）

- 树内“知识库卡片”入口
- 知识库列表 + 新建/编辑
- 知识库详情：Documents（上传/列表/删除）+ Search Playground
- 对话页：@ 圆形按钮 + 弹窗 + 输入框芯片 + 发送时携带 knowledge 选择（契约待定）

### 11.1 v0 验收标准（可测试）

- 管理：创建知识库、上传 1 个 PDF/Docx，最终状态可用，文档列表可见。
- 对话：选择 KB 后提问，回答带“引用来源”；清空 KB 后提问，行为回到普通对话。
- 文件限制：选定 1 个文件后提问，引用来源只来自该文件（允许 0 命中并提示）。
- 兼容：前端不出现任何 WeKnora session_id 或 WeKnora 专属路径字样。
